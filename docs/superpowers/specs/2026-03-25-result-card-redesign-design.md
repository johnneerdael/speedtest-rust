# Result Card Redesign Design

## Goal

Redesign the telemetry share image generated by the Rust backend so `/backend/results?id=<uuid>` returns a larger, better-spaced, branded result card that matches the Nexio portal’s dark visual style and reads cleanly when shared as a standalone image.

## Scope

This design covers:

- the image-generation logic used for telemetry share results
- visual redesign of the generated result card
- sizing, layout, typography, spacing, and footer metadata treatment
- image encoding choice for cleaner output quality
- local generation of a dummy/sample image for visual validation

This design does not cover:

- changes to telemetry collection fields or persistence format
- changes to frontend share URL generation
- changes to the stats page
- changes to the test execution flow

## Current System

The current frontend only links to the result image:

- `assets/index.html` builds `/backend/results/?id=<uuid>` and assigns it to `resultsImg`

The backend image is generated in Rust:

- route registration for `/backend/results` in `src/config/mod.rs`
- route handling in `src/http/http_server.rs` and `src/http/routes.rs`
- actual image composition in `src/results/telemetry.rs`

Today the result image is a fixed-size raster with hard-coded coordinates, minimal theming, and cramped footer spacing. The current layout makes the share result look more like a plain diagnostic snapshot than a branded social/share card.

## Architecture

The existing backend contract should remain stable:

- telemetry is still recorded through the current telemetry path
- `/backend/results?id=<uuid>` still returns a directly renderable image derived from the stored telemetry row

The redesign should stay in the rendering layer:

- fetch telemetry as today
- map the telemetry data into a redesigned card layout
- render the new card into an image buffer
- return the image through the same result route

The response type may change from JPEG to PNG if that improves crispness for text and gradients without changing the route contract.

## Visual Direction

The new result card should align with the Nexio portal, but feel stronger and more shareable than the page itself:

- dark graphite background with layered portal-style gradients
- subtle cyan, lavender, and pink glow accents
- brighter, more intentional hierarchy for metrics
- rounded blocks/panels rather than one flat undifferentiated canvas
- cleaner footer and metadata treatment

The image should feel like a designed branded card, not a screenshot.

## Layout

The selected direction is a larger version of the “balanced portal card” concept:

- branded header
- primary metric row for download and upload
- secondary metric row for ping and jitter
- footer metadata band

### Header

The header should include:

- Nexio branding or monogram treatment
- product/title text for the result card
- timestamp aligned opposite the title block

The header should feel like a compact portal top bar, not a generic watermark strip.

### Primary Metrics

Download and upload should remain the visual focus:

- each gets its own large panel or card area
- large display numerals
- clear unit placement that does not collide with the main values
- accent identity per metric, with cyan/lavender emphasis

### Secondary Metrics

Ping and jitter should be visually lighter than download/upload, but still clearly legible:

- smaller panels beneath or adjacent to the primary row
- icon treatment for quick scanning
- value and unit spacing that avoids overlap

### Footer

The footer should hold metadata in a more deliberate way:

- provider / ISP
- location or server context when available
- timestamp if not already sufficiently represented in the header
- small Nexio/LibreSpeed attribution

Footer content should never overlap with metric values.

## Data Handling

The renderer should continue using the existing telemetry row as its source. No schema changes are required for the redesign.

The implementation may normalize or trim long strings for display purposes, especially:

- ISP/provider strings
- processed location strings
- IP addresses or UUID display fragments

If a field is missing or overly long, the renderer should degrade gracefully without breaking the card layout.

## Image Encoding

The preferred output format is PNG instead of JPEG for this redesigned card because:

- text edges remain cleaner
- gradient banding is easier to control
- dark UI surfaces and thin lines hold up better

If the backend response helper currently assumes JPEG, the response path should be updated consistently with the new renderer output.

## Dummy Image Validation

The implementation must support a local validation path that renders a sample result card from hard-coded telemetry data without requiring a real stored result row.

This can be done through either:

- a dedicated dev/test helper function invoked locally, or
- a small one-off binary/test path that writes a sample image to disk

The important requirement is that the developer can generate a dummy image after implementation and inspect the real rendered output before validating against live telemetry.

## Risks And Constraints

- The current renderer relies on hard-coded coordinates, so a redesign should avoid piling more magic numbers into the same flat structure. The new layout should introduce clearer grouped regions and more explicit spacing rules.
- Long ISP/location strings can still damage layout if width constraints are not handled deliberately.
- If the response format changes to PNG, the response helper must send the correct content type.
- The backend route contract should remain stable so the existing frontend share flow continues to work.

## Acceptance Criteria

- `/backend/results?id=<uuid>` still resolves through the existing route shape
- the generated result image is visually larger and less cramped than the current version
- the card uses a dark Nexio-aligned portal style with accent color
- download/upload are clearly primary and ping/jitter are clearly secondary
- metadata/footer content no longer overlaps or crowds the main metrics
- the renderer handles missing or long strings without visibly breaking the layout
- the output format is updated consistently if PNG is adopted
- a local dummy image can be generated for visual review after implementation
